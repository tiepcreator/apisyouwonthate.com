Part of my day job is trying to get folks to write API documentation, so the
hordes of new developers joining the company on a weekly basis are not stuck
trying [guess at
contracts](https://blog.apisyouwonthate.com/guessing-api-contracts-ac1b7eaebced)
by poking around years old code. Convincing API developers to take the time to
write specs has involved dangling a myriad of carrots.

One was a continuous integration pipeline to convert their in-repo OpenAPI specs
to read-only Postman Mirrors (because who wants to maintain Postman collections
manually?!!), and some folks are excited to [use JSON Schema to offer
client-side
validation](https://blog.apisyouwonthate.com/the-many-amazing-uses-of-json-schema-client-side-validation-c78a11fbde45)
(now that we‚Äôve solved the JSON Schema &gt;&lt; OpenAPI issue.)

The carrot we‚Äôll be discussing here is contract testing: the art of validating
the shape of data provided by the API is what you think it is.

Folks are regularly asking ‚Äúhow can we keep our documentation up to date‚Äù, and
we have [covered a few solutions to
that,](https://blog.apisyouwonthate.com/keeping-documentation-honest-d9ab5351ddd4)
one of which is using JSON Schema. Unfortunately thinking of it this way leads
to responses like ‚Äúok greeeeeat, when I have finished spending a week writing up
documentation, I get to write a bunch more stuff to ensure its up to date.
Yaaaay! üôÑ‚Äù

Well that‚Äôs a fairly backwards way to think about the whole thing. I tell
people: API specifications are a _really_ powerful way to get contract testing,
and they just so happen to render as documentation too. Once you‚Äôve confirmed
your code conforms to the contract, you know your documentation is correct, so
it‚Äôs pretty much been done for free.

#### What is Contract¬†Testing

A lot of applications just check that the endpoint doesn‚Äôt entirely crap out,
which is the most basic sort of unit test:

```bash
it 'should return HTTP OK (200)' do
  get "/users/#{subject.id}"
  expect(response).to have\_http\_status(:ok)
end
```

Sure ok, that‚Äôs better than literally nothing, but we need to know what fields are there, that those fields have a specific type, and we need to make sure that doesn‚Äôt randomly change!

Some folks will add some specific checks to their unit tests, maybe something like [include_json](https://relishapp.com/waterlink/rspec-json-expectations/docs/json-expectations/include-json-matcher):

```
it "has basic info about user" do  expect(subject).to include_json(    id: 25,    email: "john.smith@example.com",    name: "John"  )end
```

Others will bump this off to BDD-style testing with Cucumber or similar tools, which is essentially doing the exact same thing:

```
Feature: User API

Scenario: Show action
    When I visit "/users/1"
    Then the JSON response at "first\_name" should be "Steve"
    And the JSON response at "last\_name" should be "Richert"
    And the JSON response should have "username"
    And the JSON response at "username" should be a string
    And the JSON response should have "created\_at"
    And the JSON response at "created\_at" should be a string
    And the JSON response should have "updated\_at"
    And the JSON response at "updated\_at" should be a string
    .... ugh it goes on and on ...
```

At this point it is essentially contract testing, but its very laborious, and is a whitelist. If a new field shows up and we‚Äôve not written a bunch of assertions for its type, we have no way of knowing if its correct. Clients might start using it, later it could change, and our test suite would never know.

Something else to consider, is that writing these assertions for each field sucks. You want your unit tests to confirm the shape of the response under all sorts of scenarios, but you do not want to copy these lines over and over again. This leads to folks shoving some assertions into the first unit test, then just crossing their fingers that other scenarios output the correct shape. ü§∑‚Äç‚ôÇÔ∏è

#### JSON Schema for ‚ÄúDRY‚Äù Contract¬†Testing

Don‚Äôt Repeat Yourself, and define reusable `schemas/components/user.json` files, populated by JSON Schema:

```json
{
  "$schema": "http://json-schema.org/draft-06/schema#",
  "title": "Foo",
  "type": "object",
  "properties": {
    "id": {
      "readOnly": true,
      "type": "string",
      "example": "123"
    },
     "uuid": {
       "type": "string",
       "format": "uuid",
       "example": "50f50f52-0d41-4a08-85ea-56423b2803c8"
     },
     "email": {
       "type": "string",
       "example": "[accountassignment8603592146@example.com](mailto:accountassignment8603592146@example.com)"
     },
     "name": {
       "type": "string",
       "example": "john.smith"
     },
     "status": {
       "type": "string",
       "example": "inactive"
     },
     "created_at": {
       "type": ["string", "null"],
       "format": "date-time",
       "example": "2018-04-09T15:45:44.358Z"
     },
  },
  "required": [
    "email",
    "name",
    "uuid"
  ]
}
```

Armed with a bunch of reusable contracts, the only thing to do is check if those contracts match the code. Ruby users can do this with [json\_matchers](https://github.com/thoughtbot/json_matchers) by Thoughtbot.

Put this config wherever you handle your RSpec setup (or check out their [alternative approaches](https://github.com/thoughtbot/json_matchers#usage)):

```ruby
require "json_matchers/rspec"

JsonMatchers.schema_root = "docs/components/schemas"
```

Then it‚Äôs simply a case of adding new tests for that context, or adding the expect line to your existing tests:

```ruby
it 'should return HTTP OK (200)' do
  get "/users/#{subject.id}"
  expect(response).to have_http_status(:ok)
end

it 'should conform to user schema' do
  get "/users/#{subject.id}"
  expect(response).to match_json_schema('user')
end
```

That‚Äôs it as far as contract testing goes. Run your tests, tweak the responses and JSON Schema, try and get it to fail. Remove a required field and it‚Äôll moan. Typecast an int to a string and it‚Äôll get sad.

If you need help creating these JSON Schema files, there‚Äôs [a few hacks to speed you up](https://blog.apisyouwonthate.com/creating-api-specifications-from-bulls-t-f5a54c005135).

#### What about¬†Docs?

Often early stage OpenAPI documentation is a lot of paths and maybe some properties, but folks often leave the schemas ‚Äúuntil later‚Äù (never).

```
responses:
    200:
      description: OK
```

With the schema missing you lose all benefit of API specifications: you have no example values, attribute tables, enums for status fields, types, etc. Luckily the new JSON Schema fill exactly that gap!

```
  responses:
    200:
      description: OK **content:
        application/json:
          schema:
            $ref: ./components/schemas/user.json**
```

_You‚Äôll need to watch out for the_ [_divergence
issue_](https://blog.apisyouwonthate.com/openapi-and-json-schema-divergence-part-1-1daf6678d86e)_,
or use Speccy to convert this JSON Schema & OpenAPI combo to ‚Äúpure OpenAPI‚Äù in
some build/CI step, but this works juuuust fine._

Then it‚Äôs just a case of [turning that OpenAPI into beautiful
documentation](https://blog.apisyouwonthate.com/turning-contracts-into-beautiful-documentation-deac7013af18).

Give it a try and let me know what you think. I really like this simple,
powerful, approach to keeping code up-to-date with your API specs.
